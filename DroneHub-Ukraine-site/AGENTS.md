# AGENTS.md

Цей файл визначає правила для всіх розробників і AI-агентів, які змінюють
DroneHub Ukraine.

## 1. Пріоритети

1. Безпека та коректність даних.
2. Зрозуміла модульна архітектура.
3. Доступність, mobile-first UX і українська локалізація.
4. Строга типізація та перевірюваність.
5. Невеликі, завершені зміни без передчасного ускладнення.

Перед реалізацією функції прочитай `PROJECT_PLAN.md` і працюй у межах поточного
етапу. Не починай наступний великий модуль, поки поточний не має тестів і не
відповідає критеріям готовності.

## 2. Мова та контент

- Основна мова користувацького інтерфейсу — українська.
- Нові тексти UI, метадані, validation/error повідомлення для користувача та
  alt-тексти пиши українською.
- Ідентифікатори коду, назви таблиць, commits і технічні логи пиши англійською.
- Не вставляй користувацький HTML без санітизації.

## 3. Архітектурні правила

- Використовуй App Router і React Server Components за замовчуванням.
- Додавай `"use client"` лише компонентам, яким справді потрібні стан, ефекти
  або браузерні API.
- Не виконуй Prisma-запити з presentation/UI-компонентів. Доступ до даних
  ізолюй у `lib/modules/*` або `lib/db`.
- Бізнес-правила розміщуй у application/domain коді, а не в JSX, middleware чи
  Prisma hooks.
- Не дублюй правила між Route Handlers, Server Actions та UI.
- Route Handlers використовуй для webhooks і зовнішніх HTTP-контрактів. Для
  внутрішніх мутацій обирай Server Actions або application services відповідно
  до контексту.
- Не додавай нову інфраструктуру або залежність без конкретної потреби та
  короткого обґрунтування.

## 4. TypeScript і стиль коду

- TypeScript працює у strict mode.
- Не використовуй `any`; для недовірених значень використовуй `unknown` і
  звуження типу.
- Не використовуй non-null assertion (`!`) без доведеної інваріанти.
- Експортовані функції та application services повинні мати зрозумілі вхідні й
  вихідні типи.
- Надавай перевагу маленьким чистим функціям і композиції.
- Не змішуй форматування та сторонній рефакторинг із функціональною зміною.
- Дотримуйся ESLint і Prettier; не вимикай правила без пояснення.

## 5. Дані та Prisma

- Схема БД і міграції є джерелом істини; не змінюй production-схему вручну.
- Кожна зміна `schema.prisma` повинна мати міграцію та перевірку на чистій БД.
- Грошові значення зберігай цілими числами в мінімальних одиницях і з валютою.
- Історичні дані замовлення зберігай snapshots, а не лише посиланнями на
  змінювані сутності.
- Для створення замовлення, резерву та зміни залишків використовуй транзакції.
- Перед додаванням спискових запитів продумай пагінацію та індекси.
- Не використовуй raw SQL з рядковою інтерполяцією неперевірених значень.
- Не видаляй історичні замовлення, платежі та аудит каскадно.

## 6. Валідація, безпека й авторизація

- Будь-які дані з форм, URL, cookies, headers, API та webhooks є недовіреними.
- Валідовуй входи на сервері спільними Zod-схемами до виконання бізнес-логіки.
- Перевіряй роль і право на конкретний ресурс на сервері для кожної захищеної
  операції. Прихований елемент UI не є авторизацією.
- Не повертай клієнту stack trace, SQL, секрети чи внутрішні деталі.
- Не логуй паролі, токени, session cookies, повні платіжні або зайві персональні
  дані.
- Додавай rate limiting до чутливих публічних endpoint-ів.
- Webhook має перевіряти підпис і бути idempotent.
- Секрети зберігай у environment variables; у репозиторії дозволений лише
  `.env.example` без реальних значень.

## 7. Помилки й логування

- Очікувані помилки моделюй типізованими кодами, наприклад
  `VALIDATION_ERROR`, `FORBIDDEN`, `OUT_OF_STOCK`.
- Користувацькі повідомлення мають бути корисними й безпечними.
- Неочікувані помилки логуй структуровано з request/correlation ID.
- Важливі адмінські зміни, auth-події, замовлення та зміни статусів записуй в
  аудит.
- Не залишай `console.log` у production-коді після введення logger abstraction.

## 8. UI, доступність і SEO

- Проєктуй mobile-first; перевіряй вузькі та широкі viewport-и.
- Використовуй семантичний HTML, підписи полів, доступні назви кнопок, видимий
  фокус і клавіатурну навігацію.
- Не передавай інформацію лише кольором.
- Зображення товарів повинні мати розміри, оптимізацію та змістовний український
  alt-текст.
- Для сторінок каталогу й товару додавай metadata, canonical URL та релевантні
  structured data.
- Не індексуй account, cart, checkout і admin.

## 9. Тести та перевірки

Для кожної зміни запускай релевантний мінімальний набір, а перед завершенням
етапу — повний набір:

```text
lint
typecheck
unit/integration tests
Playwright E2E для змінених критичних сценаріїв
production build
```

- Бізнес-правило має unit-тест.
- Репозиторій або транзакційний сценарій має integration-тест.
- Критичний користувацький потік має Playwright-тест.
- Виправлення дефекту має regression-тест, якщо це практично.
- Не послаблюй assertions лише заради проходження тесту.

## 10. Правила змін

- Спочатку досліди наявний код і локальні патерни.
- Роби одну логічну зміну за раз і тримай diff малим.
- Не реалізовуй бізнес-функції, поки не завершені базова структура, validation,
  error handling, logging і тестове налаштування.
- Оновлюй документацію, `.env.example` і міграції разом із відповідною зміною.
- Не редагуй згенеровані файли вручну.
- Не видаляй і не перезаписуй чужі незв'язані зміни.
- Позначай невирішені продуктові рішення явно; не приховуй їх випадковими
  припущеннями.

## 11. Definition of Done

Зміна готова лише коли:

- поведінка відповідає вимозі та межам поточного етапу;
- серверна валідація, авторизація й помилки опрацьовані;
- логування не розкриває чутливих даних;
- UI адаптивний і доступний;
- типи, lint, тести та build проходять;
- документація й схема БД актуальні;
- немає відомих критичних TODO, замаскованих під завершену функцію.
